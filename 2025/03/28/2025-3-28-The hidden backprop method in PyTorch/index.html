PyTorch隐晦的反向传播机制

发表于 2025-03-28 更新于 2025-04-01 分类于 Deep Learning Programming 阅读次数：
本文字数： 2.5k 阅读时长 ≈ 9 分钟

PyTorch使用基于动态图的自动微分（Autograd）机制实时记录前向操作的梯度路径，使得反向传播过程中能够按需计算梯度。这个实时记录梯度路径以及反向传播中按需计算梯度的过程被PyTorch的设计者们很好的封装隐藏起来，使得使用者们只需要搭建好前向传播模型针对模型参数设计好优化器optimizer之后再调用optimizer.zero_grad()，loss.backward()，optimizer.step()就完事了。但是只会上面的’三板斧‘还是稍微有点不足，现在让我们稍微深入了解一下PyTorch的反向传播过程。

PyTorch中的计算图

PyTorch的动态图优化使得图的构建和代码的执行同步进行也就是说每次前向传播实时生成计算图。现在我们来详细剖析一下PyTorch计算图的设计理念：该计算图的节点是操作（Function），边是张量（Tensor）。比如说我在前向传播的过程中使用了z = torch.add(x, y)，那么其对应的Function子类AddBackward~在计算图中记录torch.add的输入Tensor和反向传播所需要的信息。

上面的计算图构建过程中会出现AccumulateGrad对象，该对象对应于叶子张量的反向传播节点，就如它的命名含义那样，该对象是用于处理叶子节点的梯度累积的，通常在PyTorch中所有模型参数（model.parameters()）都是叶子张量（还记得优化器构建的过程中的torch.optim.Adam(model.parameters(), ...)和每次反向传播的时候优化器的清除梯度的操作optimizer.zero_grad()吗？😊），因为它们是手动创建的张量且它的下游没有操作（Function）节点。

计算图中插入hooks

计算图中插入hooks有什么优势呢，我们之前提及PyTorch在前向传播的过程中就已经构建好了计算图，在反向传播的过程中根据已经构建好的计算图自动计算好了模型中各个叶子张量的梯度。但是我有时候想在反向传播的过程中访问或修改梯度，那么这种自动化的过程就显得不太方便了，比如可视化热力图的Grad-CAM算法就需要访问反向传播过程中的梯度。至于修改梯度的话也有很多应用比如说我们常用的梯度裁剪操作，这种时候hooks就登场了，具体来说，它允许用户在前向传播，反向传播或梯度计算的过程中插入自定义的逻辑，可以满足我们操作或访问模型的中间输出或内部梯度的要求。

Tensor Hooks

直接注册在torch.Tensor上的钩子，用于监控或修改该张量的梯度。

Node Hooks

注册在 Node对象上的钩子，用于干预操作节点的梯度计算逻辑。

Forward Hooks

注册在 nn.Module 上的钩子，用于监控或修改模块的前向传播输出。

Backward Hooks

注册在 nn.Module 上的钩子，用于监控或修改反向传播中模块的梯度。

自定义操作（Function）

上面我们所说的都是PyTorch中的一系列已经封装好的算子在前向传播的使用过程中就在计算图中自动生成了其对应的Function子类记录一些反向传播过程中必要的东西。那为什么还要自定义操作（Function）呢？我们要明白PyTorch封装好的算子的不断拼接组合有时候并不能满足我们的需求，有些时候我们甚至要修改最底层的算子操作，比如说设计一种全新的卷积操作。这个内容难度较高，我们之后另开一篇博文来说明。
$$
X \rightarrow f_{\theta {1}}^1(X) \rightarrow f{\theta {2}}^2(f{\theta {1}}^1(X)) \rightarrow \dots \rightarrow f{\theta {n}}^n(\dots f{\theta {2}}^2(f{\theta {1}}^1(X)))\tag{1}
$$
典型前向传播过程如式$(1)$所示，其中$f{\theta {i}}^i(\cdot), i=1\dots n$表示可学习参数为$\theta{i}$的映射，一个映射中往往包含了远远多于一个操作（Function）。

常见前向传播无误反向传播错误的情况

在了解完计算图之后我们可以先了解一下backward()方法中的一些可选参数。因为前向传播无误而反向传播出现错误往往与该方法相关。

原地操作（in-place operation）

原地操作是指直接修改张量内容而不创建新的张量。一般进行原地操作的算子会带有_，比如说.add_()，.mul_()，还有python风格的+=，*=等操作。原地操作在前向传播的过程中不会出现什么问题，但在反向传播过程中有时会出现RuntimeError: a leaf Variable that requires grad is being used in an in-place operation.错误，由于叶子张量内容被篡改，导致在反向传播的时候不知道起初的叶子张量的内容来计算该张量的梯度。

梯度累积（gradient accumulation）

PyTorch的反向传播机制使得梯度在叶子张量处的梯度会随着每一次的loss.backward()而进行累积，如果不在每一次loss.backward()的之前或之后进行梯度清零的话很容易在循环的过程中发生梯度爆炸。
